# KBPrep 主设计文档：资料到知识库的质量流水线

> 本文是 KBPrep 的主设计文档。后续开发、重构、测试、验收、规则扩展、工具接入，都以本文定义的流程为准。
>
> 本文只描述目标设计，不描述当前实现状态。当前代码如果与本文冲突，应以本文作为下一步重构和补齐依据。

## 1. 项目一句话

KBPrep 是一个把各种原始资料稳定变成可进入知识库的 Markdown 的本地处理系统。

它不是单纯的格式转换工具。它的核心能力是：

1. 识别用户给的资料来源和格式。
2. 选择最合适的转换工具。
3. 先转成基础 Markdown。
4. 先验转换质量，转换不合格不能进入清洗。
5. 转换合格后，根据结构和规模决定整体处理还是分块处理。
6. 判断资料类型，选择清洗规则和保护规则。
7. 逐块清洗、逐块验收。
8. 失败时定位原因，生成修复任务和规则提案，并复跑受影响部分。
9. 全部通过后再合并、最终检查、输出交付物。
10. 用户反馈继续沉淀进规则和字典，让系统持续进化。

核心原则：

**先转好，再验转换；再分好，再清洗；逐块验收，失败就学习；全部通过，再合并发布。**

## 2. 设计目标

### 2.1 用户目标

用户只需要给出资料，不需要理解底层工具和处理细节。

系统应该能回答：

- 这个资料是什么来源？
- 这个资料应该用什么方式转换？
- 转换结果质量是否合格？
- 资料是小资料还是大资料？
- 大资料应该怎么切分？
- 这是什么类型的资料？
- 应该清洗什么？
- 必须保护什么？
- 清洗有没有误删或漏删？
- 最终结果能不能进入知识库？
- 本次经验能不能写回规则，避免下次再犯？

### 2.2 工程目标

系统必须具备：

- 可解释的处理路线。
- 可追踪的转换证据。
- 可复跑的中间结果。
- 可定位的失败原因。
- 可维护的规则和字典。
- 可验证的质量标准。
- 可扩展的工具接入方式。
- 可面向真实样本持续硬化的测试体系。

### 2.3 非目标

系统不追求：

- 所有资料百分百自动正确。
- 所有格式都自己从零实现解析。
- 用一个工具处理所有类型资料。
- 转换失败时假装成功。
- 清洗失败时只给一个失败结论。
- 用户反馈直接无审核写入长期规则。
- 为演示效果硬编码某个平台、作者、课程、品牌或路径。

## 3. 核心概念

### 3.1 原始资料

用户给系统的初始内容。可以是：

- 本地文件。
- 文件夹批量资料。
- 网页链接。
- 平台链接。
- 音频视频。
- 图片或扫描件。

### 3.2 基础 Markdown

转换阶段产出的底稿。

基础 Markdown 的目标是**尽量完整保留原始资料信息**，不是立刻清洗。它应该保留：

- 正文文字。
- 标题和章节。
- 页码或顺序线索。
- 段落关系。
- 表格。
- 代码。
- 图片引用或图片说明。
- 字幕时间轴。
- 来源证据。

### 3.3 分块

把合格的基础 Markdown 按自然结构切成适合处理的块。

分块依据包括：

- 标题。
- 章节。
- 页码。
- 段落。
- 字幕时间轴。
- 表格、代码、步骤等不可切碎的完整单元。

### 3.4 清洗规则

用于判断哪些内容应该删除、保留或待确认的规则。

清洗规则不能一刀切，必须结合：

- 通用污染模式。
- 资料类型。
- 来源特征。
- 用户已确认经验。
- 保护规则。

### 3.5 保护规则

用于防止误删正文和关键内容的规则。

必须保护：

- 正文。
- 案例。
- 方法。
- 操作步骤。
- 表格。
- 代码。
- 参数。
- 链接。
- 图片说明。
- 关键数字。
- 提示词或可复用模板。

### 3.6 质量门

每个关键阶段的硬检查。

质量门不是“失败就结束”，而是：

1. 判断是否能进入下一步。
2. 如果不能，定位原因。
3. 生成修复任务。
4. 必要时形成规则或字典提案。
5. 复跑受影响部分。

### 3.7 规则提案

质量失败或用户反馈产生的候选规则。

规则提案不能自动变成长期规则。它必须经过：

- 适用范围确认。
- 正例验证。
- 反例验证。
- 用户或维护者确认。

## 4. 总流程

完整流程如下：

1. 用户给资料。
2. 入口接收。
3. 识别来源。
4. 识别格式和基础信息。
5. 选择转换路线。
6. 转成基础 Markdown。
7. 转换质量检测。
8. 转换质量不合格时定位转换问题。
9. 转换质量合格后分析 Markdown 结构和规模。
10. 判断是小资料还是大资料。
11. 小资料整体进入内容理解。
12. 大资料按自然结构分块。
13. 判断资料类型。
14. 选择清洗规则和保护规则。
15. 逐块清洗。
16. 分块质量检查。
17. 当前分块不过关时定位失败原因。
18. 生成修复任务。
19. 生成规则或字典提案。
20. 复跑受影响分块。
21. 当前分块过关后进入合并前整体检查。
22. 检查整体是否连贯完整。
23. 不完整时基于原文证据修复衔接。
24. 完整时按原分块顺序合并。
25. 最终质量检查。
26. 最终不通过时回到失败定位。
27. 最终通过时输出交付物。
28. 用户复查。
29. 用户没有反馈，本次处理完成。
30. 用户有反馈，把反馈变成规则或字典提案。
31. 确认适用范围。
32. 做反例验证。
33. 进入长期规则库。
34. 后续同类资料自动复用。

## 5. 阶段设计

### 5.1 用户给资料

目标：

接收用户提供的原始资料，并形成待处理对象。

输入：

- 文件路径。
- 文件夹路径。
- 网页链接。
- 平台链接。
- 图片。
- 音频。
- 视频。

输出：

- 待处理对象清单。
- 来源类型初步记录。
- 输入有效性检查结果。

验收标准：

- 路径存在。
- 文件可读。
- 链接格式可识别。
- 批量资料能列出文件清单。
- 不允许静默跳过用户给出的资料。

失败处理：

- 文件不存在：反馈用户。
- 文件打不开：记录原因并反馈用户。
- 链接无法识别：进入问题说明。
- 批量资料中部分文件不可读：单独列出，不影响其他文件进入后续流程。

### 5.2 入口接收

目标：

把用户输入统一变成系统内部可管理的任务。

输入：

- 用户给的资料。

输出：

- 处理任务。
- 任务编号。
- 原始输入记录。
- 初始运行目录或过程目录。

验收标准：

- 每个任务都有唯一标识。
- 每个任务能追溯到用户原始输入。
- 批量任务能追踪每个子任务。
- 后续阶段不能丢失原始输入信息。

失败处理：

- 任务无法创建时，不进入转换。
- 写入过程目录失败时，反馈用户或维护者。

### 5.3 识别来源

目标：

判断资料来自哪里，决定后续处理方式。

来源类型：

- 本地文件。
- 文件夹批量。
- 网页链接。
- 平台链接。
- 音频视频。
- 图片或扫描件。

输出：

- 来源类型。
- 来源说明。
- 是否需要下载、提取、转写或识别。

验收标准：

- 本地资料能定位到文件。
- 文件夹能展开成清单。
- 网页链接和平台链接能区分。
- 音频视频不能被误判为普通文档。
- 图片或扫描件不能被误判为普通文本。

失败处理：

- 来源无法判断：生成问题说明。
- 链接需要登录或权限：反馈用户。
- 平台不支持：记录能力缺口。

### 5.4 识别格式和基础信息

目标：

在选择转换工具前，明确资料格式和基础风险。

需要识别：

- 文件扩展名。
- 文件大小。
- 页数。
- 图片数量。
- 音视频时长。
- 是否有可复制文本层。
- 是否可能是扫描件。
- 是否损坏。
- 是否空文件。
- 是否格式未知。

输出：

- 格式类型。
- 基础信息。
- 初步风险标签。

验收标准：

- 能直接提取文本的资料被标记出来。
- 需要图片文字识别的资料被标记出来。
- 需要字幕或语音转文字的资料被标记出来。
- 文件损坏、空文件、格式未知不能进入伪转换。

失败处理：

- 格式未知：生成问题说明。
- 文件损坏：记录原文件问题。
- 元信息不完整：允许进入保守转换，但必须在转换质量检测中重点检查。

### 5.5 选择转换路线

目标：

为不同资料选择最合适的转换路线。

转换路线包括：

- 可复制文字：直接提取。
- 办公文档：文档转换。
- 扫描件或图片：图片文字识别。
- 音频视频：提字幕或转文字。
- 网页链接：下载或抓取正文。
- 格式未知：生成问题说明。

选择原则：

- 能用成熟工具就不用自建。
- 能直接提文字就不强行图片文字识别。
- 文本层坏的文档应走图片文字识别。
- 有字幕的视频优先提字幕。
- 没字幕的音视频再转文字。
- 链接内容先变成本地材料再进入后续流程。
- 同一种格式允许多条路线，但必须记录选择理由。

输出：

- 转换路线。
- 工具选择理由。
- 工具参数。
- 预期输出。

验收标准：

- 每次转换前都有明确路线。
- 路线选择有证据。
- 不允许所有格式走同一个粗暴转换器。
- 不允许因为工具不支持就伪造成功结果。

失败处理：

- 工具缺失：反馈用户或维护者。
- 工具不支持：记录能力缺口。
- 路线选择不确定：使用保守路线，并在质量检测中提高检查要求。

### 5.6 转成基础 Markdown

目标：

把原始资料转换成可处理的基础 Markdown。

要求：

- 不在转换阶段提前清洗。
- 尽量保留原始信息。
- 尽量保留结构。
- 尽量保留来源证据。
- 记录转换过程。

输出：

- 基础 Markdown。
- 转换日志。
- 转换报告。
- 来源证据。
- 过程文件。

验收标准：

- 输出不为空。
- 标题、章节、页码、段落关系尽量保留。
- 表格、代码、图片说明、字幕时间轴尽量保留。
- 原始资料和转换结果之间能建立对应关系。

失败处理：

- 输出为空：转换质量不合格。
- 输出明显缺失：转换质量不合格。
- 过程被打断：删除坏结果并重新转换。

### 5.7 转换质量检测

目标：

确认基础 Markdown 是否有资格进入后续分块和清洗。

必须检查：

- 文字是否乱码或空白。
- 标题章节是否保留。
- 表格代码图片是否丢失。
- 页码顺序是否正常。
- 字幕时间轴是否连续。
- 输出量是否明显异常。

质量合格条件：

- 文字可读。
- 结构基本可信。
- 输出量与原始资料规模匹配。
- 关键内容没有明显丢失。
- 图片、表格、代码、字幕等特殊内容有合理保留或明确说明。

质量不合格条件：

- 大量乱码。
- 大量空白。
- 明显缺页。
- 长文档只转出极少内容。
- 表格或代码大面积丢失。
- 图片引用全部丢失且没有说明。
- 字幕时间轴断裂严重。

输出：

- 转换质量报告。
- 是否通过。
- 失败原因。
- 建议处理方式。

失败处理：

- 工具选错：换工具。
- 过程打断：删除坏结果重新转。
- 原文件质量差：记录原文件问题。
- 现有工具不支持：记录能力缺口。
- 参数不合适：调整转换参数。

硬规则：

**转换质量不合格时，不能进入清洗。**

### 5.8 定位转换问题

目标：

把转换失败从“结果不好”拆成可处理原因。

原因类型：

- 工具选错。
- 过程打断。
- 原文件质量差。
- 现有工具不支持。
- 参数不合适。

输出：

- 失败归因。
- 修复任务。
- 是否可重试。
- 是否需要反馈用户。

处理规则：

- 工具选错：回到选择转换路线。
- 过程打断：删除坏结果，回到选择转换路线。
- 参数不合适：调整参数，回到选择转换路线。
- 原文件质量差：输出问题说明和处理建议。
- 工具不支持：输出问题说明和能力缺口。

验收标准：

- 失败必须有归因。
- 可重试失败必须可复跑。
- 不可重试失败必须有用户可理解说明。

### 5.9 分析 Markdown 结构和规模

目标：

在转换质量合格后，判断资料是整体处理还是分块处理。

必须分析：

- 文字量。
- 标题层级。
- 章节关系。
- 页码关系。
- 段落和表格。
- 字幕时间轴。

判断标准：

- 小资料：整体上下文不会过大，结构简单，可以整体处理。
- 大资料：整体处理会影响清洗准确性，必须分块。
- 混合资料：可按内容结构拆成多个处理单元。

输出：

- 结构地图。
- 规模判断。
- 分块建议。

验收标准：

- 不能只按文件体积判断大小。
- 必须结合页数、时长、图片数量、文字量和结构复杂度。
- 大资料必须给出分块依据。

### 5.10 判断小资料还是大资料

目标：

决定后续处理策略。

小资料处理：

- 整体进入内容理解。
- 整体判断资料类型。
- 整体选择规则。
- 整体清洗后仍要做质量检查。

大资料处理：

- 先按自然结构分块。
- 每块单独进入内容理解。
- 每块单独清洗。
- 每块单独质量检查。
- 最后再合并。

验收标准：

- 大资料不能直接整体清洗。
- 小资料不能被无意义切碎。
- 切分大小必须适合处理和验收。

### 5.11 按自然结构分块

目标：

把大资料切成大小合适、上下文完整、可追踪的处理单元。

分块方式：

- 按标题分块。
- 按章节分块。
- 按页码分块。
- 按段落分块。
- 按字幕时间轴分块。
- 保护表格、代码、步骤不被切碎。

输出：

- 分块清单。
- 每块来源范围。
- 每块顺序。
- 每块标题路径。
- 每块页码或时间范围。
- 不可切碎单元记录。

验收标准：

- 分块不能过大。
- 分块不能过小。
- 表格、代码、操作步骤、案例不能被硬切断。
- 每块都能追溯回原始资料。
- 后续合并能按原顺序恢复。

失败处理：

- 切坏上下文：回到结构分析。
- 分块过大：调整切分策略。
- 分块过小：合并相邻块。
- 切断保护内容：修复保护规则。

### 5.12 内容理解和资料类型判断

目标：

判断资料或分块属于什么内容类型，为规则选择提供依据。

资料类型：

- 课程资料。
- 报告资料。
- 网页文章。
- 电子书。
- 访谈或转录稿。
- 代码或技术资料。
- 混合资料或未知资料。

输出：

- 资料类型。
- 判断理由。
- 置信度。
- 是否需要保守清洗。

验收标准：

- 课程资料要保护步骤、案例、方法。
- 报告资料要保护表格、数字、结论和来源。
- 网页文章要区分正文和页面包装。
- 电子书要保护章节和连续性。
- 转录稿要处理口语重复，但不能删掉有效观点。
- 技术资料要保护代码、参数、配置和命令。
- 混合资料不能强行归为单一类型。

失败处理：

- 类型不确定：降低清洗力度，进入待确认。
- 类型判断错误：进入质量回路，补分类信号。

### 5.13 选择清洗规则和保护规则

目标：

根据资料类型、来源和用户经验，选择本次清洗规则组合。

规则来源：

- 通用清洗规则。
- 资料类型规则。
- 来源专属规则。
- 用户已确认规则。
- 保护规则。

规则选择必须回答：

- 这次应该删除什么？
- 这次必须保护什么？
- 哪些内容需要待确认？
- 哪些规则只对特定来源生效？
- 哪些规则可能误伤？

验收标准：

- 规则组合可解释。
- 规则来源可追踪。
- 保护规则优先级高于删除规则。
- 未确认用户反馈不能直接成为长期规则。
- 不允许把平台、作者、课程、品牌硬编码进程序逻辑。

失败处理：

- 规则太弱：生成删除规则提案。
- 规则太猛：生成保护规则提案。
- 规则冲突：进入待确认。
- 来源特殊：生成来源专属规则提案。

### 5.14 逐块清洗

目标：

对资料块执行清洗，同时保护正文和关键内容。

可删除内容：

- 广告包装。
- 二维码。
- 入群引导。
- 重复页眉页脚。
- 无意义口播。
- 明显重复的营销话术。

必须保留内容：

- 正文。
- 案例。
- 方法。
- 步骤。
- 表格。
- 代码。
- 图片说明。
- 关键数字。
- 链接。
- 参数。

输出：

- 保留内容。
- 删除内容。
- 待确认内容。
- 清洗原因。
- 保护原因。

验收标准：

- 每个删除都能解释原因。
- 每个保护都能解释原因。
- 删除内容有清单。
- 待确认内容有清单。
- 清洗不能破坏块内结构。

### 5.15 分块质量检查

目标：

确认每个清洗后的分块是否可靠。

必须检查：

- 有没有误删正文。
- 有没有广告残留。
- 有没有表格代码断裂。
- 有没有步骤缺失。
- 有没有上下文被切坏。
- 有没有规则误伤。

输出：

- 分块质量结论。
- 失败原因。
- 证据。
- 下一步动作。

通过条件：

- 正文完整。
- 噪音已清理到可接受范围。
- 结构没有破坏。
- 保护内容没有误删。
- 待确认内容明确列出。

不过关处理：

- 定位失败原因。
- 生成修复任务。
- 生成规则或字典提案。
- 复跑受影响分块。

硬规则：

**不是全部分块通过，就不能进入合并。**

### 5.16 定位失败原因

目标：

把分块质量失败拆成可修复原因。

失败类型：

- 转换问题。
- 切分问题。
- 资料类型判断问题。
- 清洗规则太弱。
- 清洗规则太猛。
- 字典覆盖不足。

处理方式：

- 转换问题：回到转换质量回路。
- 切分问题：回到结构分析和分块策略。
- 类型判断问题：补分类信号。
- 规则太弱：生成删除规则提案。
- 规则太猛：生成保护规则提案。
- 字典覆盖不足：生成字典补充提案。

验收标准：

- 失败原因不能只写“质量不合格”。
- 每个失败必须指向一个可执行动作。
- 可复跑问题必须只复跑受影响部分。

### 5.17 生成修复任务

目标：

把失败原因转成后续可执行任务。

修复任务包括：

- 换转换工具。
- 调整转换参数。
- 重做分块。
- 修改资料类型判断。
- 补清洗规则。
- 补保护规则。
- 补字典。
- 交给用户确认。

输出：

- 修复目标。
- 受影响分块。
- 修复方式。
- 复跑范围。
- 验收标准。

验收标准：

- 任务必须能执行。
- 任务必须有目标。
- 任务必须有复查方式。
- 不能只写“优化规则”这种空话。

### 5.18 生成规则或字典提案

目标：

把本次处理经验沉淀为可维护规则。

提案类型：

- 删除规则提案。
- 保护规则提案。
- 分类规则提案。
- 来源专属规则提案。
- 字典补充提案。

提案必须包含：

- 触发样本。
- 问题说明。
- 建议规则。
- 适用范围。
- 可能误伤范围。
- 正例。
- 反例。
- 是否需要用户确认。

验收标准：

- 没有证据不能生成长期规则。
- 没有适用范围不能接受。
- 没有反例验证不能进入长期规则库。

### 5.19 复跑受影响分块

目标：

只对受影响部分复跑，验证修复是否有效。

输入：

- 受影响分块。
- 修复任务。
- 临时规则或参数。

输出：

- 复跑结果。
- 修复是否有效。
- 新旧结果对比。

验收标准：

- 复跑范围不能扩大到无关资料。
- 复跑结果必须再次进入质量检查。
- 修复不能引入新误删或新残留。

### 5.20 合并前整体检查

目标：

在全部分块通过后，确认它们能安全合并。

检查项：

- 分块顺序。
- 章节连续性。
- 前后衔接。
- 内容是否缺失。
- 是否需要证据内修复。

输出：

- 可合并结论。
- 衔接问题清单。
- 缺失问题清单。

验收标准：

- 分块顺序与原始结构一致。
- 章节没有断裂。
- 前后内容不割裂。
- 没有因为清洗导致内容缺口。
- 合并前问题必须先解决。

### 5.21 基于原文证据修复衔接

目标：

修复分块之间的轻微割裂，让最终 Markdown 完整、通顺、可读。

允许修复：

- 标题衔接。
- 段落过渡。
- 重复口播整理。
- 前后引用关系整理。
- 分块边界造成的轻微断裂。

禁止修复：

- 凭空补充原文不存在的知识。
- 自行添加事实、案例、数据。
- 用模型猜测缺失内容。
- 把不确定内容写成确定结论。

验收标准：

- 所有补充都能回到原文证据。
- 无证据内容必须进入待确认。
- 修复后要再次做整体检查。

### 5.22 按原分块顺序合并

目标：

把通过质量检查的分块合并成最终候选 Markdown。

要求：

- 按原分块顺序合并。
- 保留标题层级。
- 保留必要来源信息。
- 保留图片、表格、代码和链接关系。
- 保留分块记录。

输出：

- 最终候选 Markdown。
- 合并记录。

验收标准：

- 怎么切开的，就能怎么解释合回去。
- 合并后结构清晰。
- 合并后不丢来源。
- 合并后不重复堆叠标题或段落。

### 5.23 最终质量检查

目标：

发布前做最后验收。

检查项：

- 正文完整。
- 噪音清除。
- 结构通顺。
- 来源可追踪。
- 规则改进已记录。

通过条件：

- 所有关键正文保留。
- 明显广告和包装已清除。
- 表格、代码、步骤、关键数字没有误删。
- 图片和链接关系合理。
- 删除内容有证据。
- 待确认内容有清单。
- 质量失败经验已形成提案或任务。

不过关处理：

- 回到失败定位。
- 不允许发布最终结果。

### 5.24 输出交付物

目标：

给用户的不只是最终 Markdown，还包括质量证据和后续改进入口。

交付物包括：

- 最终 Markdown。
- 被删除内容清单。
- 待确认内容清单。
- 质量报告。
- 分块记录。
- 规则改进建议。

验收标准：

- 最终 Markdown 可读、可编辑、可进入知识库。
- 删除内容可追踪。
- 待确认内容明确。
- 质量报告能解释本次结果是否可靠。
- 分块记录能追溯处理过程。
- 规则建议能进入后续维护流程。

### 5.25 用户复查

目标：

让用户的判断进入系统，而不是只停留在聊天里。

用户反馈类型：

- 这里没清干净。
- 这里误删了。
- 这种内容以后要删。
- 这种内容以后不能删。
- 这种资料应该归为另一类。
- 这种来源有特殊包装。

输出：

- 用户反馈记录。
- 规则或字典提案。
- 适用范围候选。

验收标准：

- 用户反馈必须绑定到具体内容或具体模式。
- 不能把模糊反馈直接写成长期规则。
- 反馈必须能复查。

### 5.26 确认适用范围

目标：

明确一条规则应该在哪里生效，防止误伤其他资料。

适用范围维度：

- 资料类型。
- 来源。
- 污染模式。
- 保护模式。
- 语言。
- 文件格式。
- 是否批量任务。

输出：

- 规则适用范围。
- 不适用范围。
- 风险说明。

验收标准：

- 没有适用范围的规则不能进入长期规则库。
- 来源专属规则不能变成全局规则。
- 删除规则必须有保护反例。

### 5.27 反例验证

目标：

防止一次误判变成长期污染。

验证方式：

- 用同类正例验证规则有效。
- 用不同资料反例验证规则不误伤。
- 用保护内容样本验证不会误删正文。
- 用真实样本复跑验证。

输出：

- 反例验证结果。
- 是否允许进入长期规则库。

验收标准：

- 反例不过关，规则不能接受。
- 验证样本不足，规则只能保持提案状态。
- 接受规则必须可撤回或可禁用。

### 5.28 进入长期规则库

目标：

把确认有效的经验变成系统长期能力。

长期规则必须记录：

- 规则内容。
- 规则类型。
- 适用范围。
- 来源证据。
- 正例。
- 反例。
- 创建原因。
- 创建时间。
- 是否可禁用。

验收标准：

- 长期规则可追踪。
- 长期规则可测试。
- 长期规则可回滚。
- 后续同类资料能自动复用。

## 7. 质量门设计

### 7.1 转换质量门

位置：

基础 Markdown 生成之后，结构分析之前。

目的：

防止坏底稿进入清洗。

必须拦截：

- 空输出。
- 乱码。
- 明显缺页。
- 输出量异常。
- 表格、代码、图片、字幕大面积丢失。
- 转换过程被打断。

不通过动作：

- 换工具。
- 调参数。
- 删除坏结果重转。
- 记录原文件问题。
- 记录能力缺口。
- 反馈用户。

### 7.2 分块质量门

位置：

逐块清洗之后，合并之前。

目的：

防止有问题的分块进入最终合并。

必须拦截：

- 正文误删。
- 广告残留。
- 表格代码断裂。
- 步骤缺失。
- 上下文被切坏。
- 规则误伤。

不通过动作：

- 定位失败原因。
- 生成修复任务。
- 生成规则或字典提案。
- 复跑受影响分块。

### 7.3 合并前质量门

位置：

全部分块通过之后，合并之前。

目的：

防止分块之间割裂、错序、缺失。

必须拦截：

- 分块顺序错误。
- 章节不连续。
- 前后不衔接。
- 内容缺失。
- 证据不足的补写。

不通过动作：

- 基于原文证据修复衔接。
- 回到分块策略。
- 进入待确认。

### 7.4 最终质量门

位置：

合并后，发布前。

目的：

确认最终交付物可以进入知识库。

必须拦截：

- 正文不完整。
- 噪音未清除。
- 结构不通顺。
- 来源不可追踪。
- 删除证据缺失。
- 质量失败经验未记录。

不通过动作：

- 回到失败定位。
- 不发布最终结果。

## 8. 工具接入原则

工具接入遵循以下原则：

1. 优先使用成熟工具。
2. 能直接使用就不要自建。
3. 同一类型允许多工具路线。
4. 工具选择必须有理由。
5. 工具失败必须能归因。
6. 工具输出必须进入质量检测。
7. 工具能力缺口必须记录。
8. 工具不能绕过主流程直接写最终结果。

不同类型资料的工具路线：

- 普通文本：直接提取。
- 办公文档：文档转换工具。
- PDF：先判断文本层质量，再决定直接提取或图片文字识别。
- 图片和扫描件：图片文字识别。
- 音频视频：先提字幕，没有字幕再转文字。
- 网页链接：下载或抓取正文。
- 平台链接：先变成本地材料，再进入主流程。

## 9. 规则和字典设计

### 9.1 规则分类

规则分为：

- 通用清洗规则。
- 资料类型规则。
- 来源专属规则。
- 用户确认规则。
- 保护规则。

### 9.2 字典分类

字典分为：

- 广告词字典。
- 二维码和入群引导字典。
- 页眉页脚模式字典。
- 重复口播字典。
- 保护内容字典。
- 来源专属模式字典。
- 资料类型信号字典。

### 9.3 规则生命周期

规则生命周期：

1. 发现问题。
2. 生成提案。
3. 绑定证据。
4. 确认适用范围。
5. 正例验证。
6. 反例验证。
7. 用户或维护者确认。
8. 进入长期规则库。
9. 后续复用。
10. 定期复查或回滚。

### 9.4 规则升级限制

以下规则不能直接升级：

- 没有证据的规则。
- 只有一次模糊反馈的规则。
- 没有适用范围的规则。
- 没有反例验证的规则。
- 可能误伤正文的删除规则。
- 针对单一来源却写成全局的规则。

## 10. 失败回路设计

系统必须把失败当成学习入口。

失败回路包括：

1. 发现失败。
2. 定位失败原因。
3. 生成修复任务。
4. 生成规则或字典提案。
5. 复跑受影响部分。
6. 再次质量检查。
7. 通过后继续流程。
8. 不通过则继续归因或进入待确认。

失败不能只记录“失败”。必须说明：

- 哪个阶段失败。
- 失败证据是什么。
- 失败原因是什么。
- 是否可自动修复。
- 是否需要用户确认。
- 是否产生规则提案。
- 是否已经复跑验证。

## 11. 用户反馈设计

用户反馈是系统长期进化的一部分。

用户反馈必须进入以下流程：

1. 记录反馈。
2. 绑定原文或结果位置。
3. 判断反馈类型。
4. 生成规则或字典提案。
5. 确认适用范围。
6. 做反例验证。
7. 确认后进入长期规则库。

用户反馈不能直接改写全局规则。

反馈类型：

- 删除不足。
- 误删正文。
- 分类错误。
- 来源模式未识别。
- 分块不合理。
- 合并不通顺。
- 输出格式不适合知识库。

## 12. 验收标准

### 12.1 单文件验收

单文件处理通过必须满足：

- 资料来源和格式识别正确。
- 转换路线可解释。
- 基础 Markdown 通过转换质量检测。
- 大文件按自然结构分块。
- 资料类型判断合理。
- 清洗规则和保护规则选择合理。
- 每个分块通过质量检查。
- 合并前整体检查通过。
- 最终质量检查通过。
- 输出交付物完整。
- 如有失败，失败原因和修复任务明确。
- 如有用户反馈，反馈进入规则提案。

### 12.2 批量处理验收

批量处理通过必须满足：

- 每个文件都有独立处理状态。
- 失败文件不会污染成功文件。
- 批量摘要能说明成功、失败、待确认数量。
- 每个失败都有原因。
- 每个成功都有质量报告。
- 批量任务可以从失败点继续。

### 12.3 新格式接入验收

新增格式必须满足：

- 能识别来源和格式。
- 能选择转换路线。
- 能生成基础 Markdown。
- 有转换质量检测。
- 有真实样本测试。
- 有失败样本测试。
- 有回退或反馈机制。

### 12.4 新清洗规则验收

新增清洗规则必须满足：

- 有正例。
- 有反例。
- 有适用范围。
- 有保护规则。
- 有复跑验证。
- 不误伤正文。
- 可禁用或可回滚。

### 12.5 新工具接入验收

新增工具必须满足：

- 工具选择理由明确。
- 输入输出可控。
- 失败可归因。
- 输出进入质量检测。
- 不绕过主流程。
- 有真实样本验证。

## 13. 开发实施拆分

后续开发应按以下能力拆分，不应混在一个大改动里。

### 13.1 入口和来源识别能力

实现内容：

- 本地文件接收。
- 文件夹批量接收。
- 链接接收。
- 图片音频视频识别。
- 输入记录。

验收：

- 各来源都能生成任务。
- 无效输入能给用户明确说明。

### 13.2 格式识别和转换路线能力

实现内容：

- 格式识别。
- 基础信息提取。
- 转换路线选择。
- 工具选择理由记录。

验收：

- 不同格式能进入不同路线。
- 格式未知能生成问题说明。

### 13.3 转换和转换质量门能力

实现内容：

- 基础 Markdown 生成。
- 转换报告。
- 转换质量检测。
- 转换失败归因。
- 重转机制。

验收：

- 转换不合格不能进入清洗。
- 可重试问题能重转。
- 不可重试问题能反馈用户。

### 13.4 结构分析和分块能力

实现内容：

- Markdown 结构分析。
- 小资料大资料判断。
- 自然结构分块。
- 分块记录。

验收：

- 大资料能合理分块。
- 保护内容不被切碎。
- 分块可追踪、可合并。

### 13.5 内容类型判断能力

实现内容：

- 资料类型识别。
- 混合类型处理。
- 置信度和理由记录。

验收：

- 不同资料类型能选择不同规则。
- 不确定类型能保守处理。

### 13.6 清洗和保护能力

实现内容：

- 规则组合。
- 逐块清洗。
- 保护规则。
- 删除和待确认清单。

验收：

- 正文不误删。
- 噪音可清除。
- 删除和保护都有原因。

### 13.7 分块质量检查和失败迭代能力

实现内容：

- 分块质量检查。
- 失败归因。
- 修复任务。
- 规则和字典提案。
- 受影响分块复跑。

验收：

- 失败能定位。
- 修复能复跑。
- 经验能沉淀。

### 13.8 合并和最终质量能力

实现内容：

- 合并前检查。
- 证据内修复。
- 按原顺序合并。
- 最终质量检查。

验收：

- 合并后完整、通顺、可追踪。
- 不通过不能发布。

### 13.9 用户反馈和长期规则库能力

实现内容：

- 用户反馈记录。
- 规则提案。
- 适用范围确认。
- 反例验证。
- 长期规则入库。

验收：

- 用户反馈可复用。
- 长期规则可测试、可回滚。

## 14. 测试策略

### 14.1 样本类型

测试样本必须覆盖：

- 普通文本。
- Markdown 笔记。
- PDF。
- 扫描 PDF。
- 图片。
- Word 文档。
- 表格文档。
- 演示文档。
- 网页保存件。
- 音频。
- 视频。
- 字幕。
- 电子书。
- 代码资料。
- 混合资料。

### 14.2 质量场景

必须测试：

- 转换成功。
- 转换失败。
- 工具选错。
- 过程打断。
- 原文件质量差。
- 大文件分块。
- 小文件整体处理。
- 清洗误删。
- 清洗漏删。
- 广告残留。
- 表格断裂。
- 代码断裂。
- 步骤缺失。
- 合并不连贯。
- 用户反馈进入规则提案。

### 14.3 回归测试

每次新增规则、字典或工具，都必须回归：

- 正例是否通过。
- 反例是否不误伤。
- 旧样本是否不退化。
- 失败是否仍能归因。

## 15. 开发约束

必须遵守：

- 不硬编码具体平台、作者、品牌、课程到程序逻辑。
- 不用假数据演示完整流程。
- 不跳过转换质量检测。
- 不跳过分块质量检查。
- 不把质量失败当作简单停止。
- 不把用户反馈直接写入长期规则。
- 不绕过主流程直接输出最终结果。
- 不把所有资料强行走同一个工具。
- 不在转换阶段提前清洗。
- 不凭空补充原文没有的内容。

## 17. 完成定义

一个功能、工具、规则或流程改动，只有同时满足以下条件才算完成：

1. 符合本文主流程。
2. 不绕过任何质量门。
3. 有输入、输出和过程记录。
4. 有失败归因。
5. 有可执行修复方式。
6. 有测试样本。
7. 有验收结果。
8. 有规则或字典影响说明。
9. 不污染无关资料类型。
10. 文档和流程图同步更新。
