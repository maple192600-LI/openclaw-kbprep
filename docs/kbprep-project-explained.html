local CLI · Python worker · Obsidian Markdown

把乱资料变成可检查的知识库材料

KBPrep 的核心目标很简单：你给它一个本地文件，它先判断文件类型，再转换成 Markdown，再检查有没有漏正文、漏图片、漏表格、误删案例，最后才允许发布成 Obsidian 或普通 Markdown。

这不是聊天机器人，也不是 RAG 系统。它是一条本地文件清洗流水线，负责把资料整理到“能进知识库”的状态。

一、这个项目到底是什么

用大白话说，KBPrep 是一个本地命令行工具，专门把 PDF、Word、PPT、网页、字幕、Markdown、EPUB、代码文件这些资料，转换成干净、可追溯、适合 Obsidian 或大模型知识库使用的 Markdown。

不是做什么

它不是平台插件仓库

当前方向是 host-neutral，也就是不把 Claude Code、Codex、OpenClaw、Hermes 这些宿主的业务逻辑写进核心项目。不同宿主要用它，就调用 CLI。

真正做什么

它做文件到 Markdown 的质量闭环

不是“转一下就完事”，而是转换、清洗、检查、记录证据、失败时阻止发布。

给谁用

给本地知识库整理流程用

特别适合把课程资料、报告、电子书、网页保存件、会议字幕、财务资料、代码说明整理成可长期维护的知识材料。

二、它是怎么设计的

项目分成三层：Node/TypeScript 负责命令入口和打包，Python worker 负责真实转换和质量判断，rules 负责可维护的清洗规则。

TypeScript 层

主要在 src/。它提供 CLI 命令、运行 Python worker、处理超时、打包成 npm 可执行命令。

kbprep-analyze
kbprep-prepare
kbprep-apply-review
kbprep-feedback
kbprep-cleanup
kbprep-batch

Python worker 层

主要在 python/kbprep_worker/。它负责格式诊断、转换、切块、清洗、质量门禁、Obsidian 输出、反馈学习。

diagnose/
converters/
quality/
feedback/
obsidian_kb/
stages/pipeline_core.py

规则层

主要在 rules/。清洗知识放在 JSON 规则里，而不是写死在 Python 代码里。这样后续要改“什么算广告、什么该保留”，不用到处改代码。

文档和门禁层

docs/ 说明能力边界，scripts/ 放发布检查脚本，src/test/ 和 python/tests/ 放测试。

三、它怎么工作

一份文件从进入 KBPrep 到最终输出，大致会经历下面这些步骤。

先诊断文件

判断是 Markdown、PDF、Office、EPUB、HTML、字幕、代码，还是当前不支持的媒体文件。PDF 还会继续判断：文字层好不好、是不是扫描件、是否需要 OCR。

选择转换路线

普通文本直接转，Office XML 本地解析，EPUB 解析 XHTML，PDF 根据诊断结果走文字层或 MinerU/OCR。

转换成中间 Markdown

生成 converted.md，并尽量保留标题、段落、表格、代码、链接、图片引用、页码证据。

切成可追踪的块

生成 blocks.jsonl。每一块都带来源信息，方便后续知道“这段话从哪里来、为什么被删、删得对不对”。

按规则清洗

删除广告、二维码、扫码入群、页眉页脚、重复口播、无关包装内容；同时保护教程步骤、参数、链接、代码、表格、案例细节。

跑质量门禁

如果发现图片缺失、正文细节被误删、表格丢了、代码块坏了、污染内容还残留，就不发布最终文件，只留下诊断材料。

通过后才发布

普通模式发布源文件旁边的 Markdown；Obsidian 模式发布一个 Obsidian 文件夹。失败时不会更新 latest.json，避免把坏结果当成最新成果。

四、它有哪些能力和效果

这里按当前能力矩阵讲，不能夸大。verified 是已经比较稳，partial 是能处理但不承诺复杂场景全覆盖，unsupported 是明确不支持。

资料类型 当前状态 怎么处理 效果 风险边界
Markdown / TXT / CSV / TSV verified 直接读取文本和 Markdown 结构 保留标题、段落、表格、链接、代码细节 清洗规则太宽时仍可能误删，所以有质量门禁
代码 / 配置文件 verified 包进 Markdown 代码块 不总结代码，保留参数、注释、URL、命令 代码不能按普通文章清洗
字幕 / transcript verified 转成可读 transcript 保留语序，清理明显口水和时间戳噪声 字幕噪声还需要按文档类型继续优化
HTML / 本地网页 partial 提取可见文字、标题、列表、链接、图片引用 能把保存网页变成可读 Markdown 导航栏、页脚、广告壳需要规则继续清
DOCX / PPTX / XLSX partial 解析 Office XML 保留文本、幻灯片顺序、表格、部分图片 复杂版式、图表、复杂表格还不能保证完整
EPUB partial 解析 XHTML 和 spine 顺序 适合电子书章节转 Markdown 脚注、复杂表格、自定义 XHTML 还需要更多样本
PDF partial 先诊断文字层；可信则直接提取，不可信则走 OCR 文本型 PDF 快，扫描件可保留 OCR 路线证据 复杂排版、坏文字层、扫描质量差时必须看质量报告
图片 / 音视频 / 旧 Office / MOBI unsupported 不假装支持 会明确告诉你需要外部转换或先转 transcript 不能宣传成 v1 已支持

五、输出文件怎么看

KBPrep 的重点是“能追责”。不是只给你一个 cleaned.md 就完事。

cleaned.md

清洗后的正文过程文件。能读，但不能单独作为验收依据。

discarded.md

被删除的内容。要检查这里有没有误删正文、案例、方法、参数。

review_needed.md

系统不敢自动决定的内容。发布前应该人工看一眼。

quality_report.json

质量门禁结果。strict error 不为空，就不能当作最终结果。

conversion_report.json

记录实际转换路线，尤其 PDF 是否走文字层、是否 fallback 到 OCR。

latest.json

只有成功发布才应该更新。它告诉你最终可用成果在哪里。

六、反馈学习怎么防止跑偏

用户说“以后删这种东西”时，KBPrep 不会立刻改规则，而是先生成可审核的规则提案。

为什么不能直接学习

因为一句反馈可能太宽，比如“删掉关注公众号”。如果正文是在讲公众号运营方法，这句话就不能删。所以系统会保留例子和反例，要求确认。

正确流程

用户反馈
生成 proposed_rules.jsonl
人工确认
写入 accepted_rules.jsonl
重新跑原文档验证
通过后才影响后续清洗

七、怎么判断它能发布

现在项目已经有两个关键门禁：一个检查代码和包，一个拿真实 Obsidian 文档跑烟测。

发布总检查

npm run release:check

它会跑依赖锁、空白检查、build、dist 同步检查、TypeScript 测试、Python lint、Python coverage、审计脚本、打包检查、npm 安全审计。

真实 Vault 烟测

npm run vault:smoke

它从 F:\Obsidian-Vault 复制代表样本到临时目录测试，覆盖 Markdown、PDF、PPTX、DOCX、EPUB、HTML、VTT、TXT、unsupported 媒体和 batch。

重要提醒

真实 Vault 测试必须先复制到临时目录，不能直接对原 Vault 文件跑 kbprep-prepare，因为标准模式会把最终 Markdown 发布到输入文件旁边。

不能吹过头的地方

当前 Python coverage 只是达到 30% 门槛，不是高覆盖。PDF、Office、EPUB 仍是 partial，表示能处理一部分真实场景，但不能承诺所有复杂文件都完美。
