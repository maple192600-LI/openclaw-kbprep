01 / executive summary

一句话状态

KBPrep 现在已经是一个可运行、可验证、已加载到 OpenClaw 的本地知识库预处理工具；但默认 curated_obsidian_kb 的最终交付物和 cleanup finalize 的收尾逻辑还存在口径断点。

已经从“插件小工具”推进到“可用的知识库预处理工具”，但还不是完全闭环产品。

当前主线不是 RAG、不是远程采集、不是总结改写，而是把本地杂乱资料转成可审计、可保留细节、可进入 Obsidian 或 LLM Wiki 的知识库材料。

已成立

核心能力能跑通

单文件 prepare、批量 prepare、OpenClaw 插件、独立 CLI、Python worker、质量门、审计文件、Obsidian 输出都已经存在。

仍需闭合

默认交付链路不够顺

默认 curated 模式下 final_md=null，但单文件 cleanup 仍强制要求 final_md，会影响“处理完成后清理”的正常链路。

产品边界

准备工具，不是知识系统全部

它负责本地材料转换、清洗、策展、审计；不负责 RAG index、远程平台下载、音视频自动转写和完整知识库运营。

02 / what happened

过程复盘

这个项目真正的开发过程，不是简单加几个工具，而是反复把“工程产物”纠正成“人真的能用的资料处理流程”。

初始目标

把本地原材料变成可进入知识库的干净 Markdown

目标不是做一个炫技插件，而是服务 Obsidian / LLM Wiki：尽量无损转换，保留细节，清掉污染，留下可追溯证据。

第一次纠偏

输出不能扔在内部目录

你明确纠正过：源文件在哪里，处理后的同名 Markdown 就应该在哪里。output_root 应该是审计/过程材料，不是用户找结果的主入口。

第二次纠偏

过程产物不能无限堆积

后来补了 kbprep_cleanup：finalize、expired、all 三个动作，用来把接受结果后的临时材料清掉。

质量纠偏

知识库不是营销稿

你指出输出里仍有大量生财有术包装内容。这个反馈推动项目从“普通 Markdown 清洗”升级为 curated_obsidian_kb 知识库策展。

加固阶段

0.5.1：主干能力稳定，但默认交付仍需闭合

当前已改名为 npm 包 kbprep，OpenClaw adapter id 保留 openclaw-kbprep；测试、打包、插件验证通过。

03 / repeated user requirements

你反复强调的问题

这些要求不是“偏好”，而是这个项目的产品原则。后续任何 Claude Code 或 Codex 接手，都必须按这些原则验收。

结果位置要符合人的直觉

源文件在哪里，最终结果就应该能在那里或以非常明确的最终目录被拿到。用户不应该去翻内部 run folder。

过程产物必须有生命周期

runs/、converted.md、blocks.jsonl、quality_report.json 是过程材料，不应该永久堆积。

知识库不是营销资料排版

要保留正文、案例、观点、方法、步骤、参数、工具、提示词、失败经验；要剥离扫码、福利、作者光环、社群包装。

不能假闭环

不能因为 cleaned.md 看着整齐就宣布成功；必须看质量报告、discarded、review_needed 和实际运行结果。

运行时验证也是完成标准

改插件不能只改源码，还要验证 OpenClaw Gateway 真的加载了新版本，工具名和 schema 真的在 runtime 中出现。

文档、测试、工具 schema 要一致

不能代码是一套、README 一套、skill 一套、测试只覆盖旧模式。AI Coding 工具最容易在这种地方继续跑偏。

04 / mistakes and traps

踩过的坑

下面这些问题都在项目推进中出现过，后面做任何重构或功能扩展，都应该拿它们当回归清单。

坑 错误原因 现在的处理方式 后续风险
把 output_root 当主交付位置 站在工程目录视角，而不是用户拿结果的视角。 标准模式已支持源文件旁发布 source-stem.md 或 *.cleaned.md。 curated 模式现在改以 obsidian/ 为交付物，文档和 cleanup 需要同步。
没有一开始设计 cleanup 生命周期 只想着“转换成功”，没有想着“用户验收后如何收尾”。 已新增 kbprep_cleanup(finalize/expired/all)。 默认 curated 模式下 finalize 仍可能因 final_md=null 卡住。
清洗目标太弱 只做 Markdown 清洁，没有做知识库策展。 默认 profile 改为 curated_obsidian_kb，剥离作者/营销/包装噪音。 仍要继续扩充不同来源、语言和行业资料的规则。
误删细节块 把含有营销词的真实方法块误判成 marketing_cta。 质量门现在检查 detail-bearing blocks，操作步骤、提示词、代码、表格不能随便丢。 新来源里仍可能出现“营销词 + 方法内容”混合段，需要审计 discarded。
PDF 坏文本层被错误当最终质量 OCR 已经替代坏文本层，但质量门还拿坏文本层打分。 已区分 rejected source text layer 和 post-convert output quality。 更多转换器接入后，也要保留“被替代源质量”和“最终交付质量”的区别。
AI review 过早退出 遇到 strict-quality failure 就直接停止，导致可恢复路径跑不到。 已允许 curation-heavy / OCR fallback 场景继续进入 review。 非 OpenClaw provider backend 仍未完整实现。
版本和运行时不同步 JS、OpenClaw manifest、Python worker 版本没有统一维护。 现在 0.5.1 已统一，并通过 runtime inspect 证实加载。 每次 release 都必须检查 source、dist、package、runtime 四层。

06 / project map

项目地图

这个项目现在是三层结构：OpenClaw/CLI 入口、TypeScript adapter 和 Python worker core。真正的业务复杂度集中在 Python worker 的转换、清洗、策展和质量门。

用户入口 OpenClaw tools、standalone CLI、未来其他 host adapter。
TypeScript adapter src/adapters/openclaw/index.ts 注册工具 schema 和超时配置。
Worker bridge src/worker.ts spawn Python CLI，处理 stdin、stdout JSON envelope、stderr 日志。
Python CLI python/kbprep_worker/cli.py 分发 preflight、diagnose、prepare、cleanup、batch。
Pipeline core prepare.py 负责转换、normalize、blockify、清洗、策展、质量门。
Artifacts output_root、runs/、obsidian/、审计文件和最终交付物。
文件 / 模块 承担责任 为什么重要
README.md 项目定义、用户流程、运行时说明、工具边界。 现在存在 curated 交付物和源文件旁 Markdown 的口径冲突，需要修正。
openclaw.plugin.json OpenClaw 插件身份、版本、config schema、工具 contract。 OpenClaw runtime 是否识别插件，靠这里和 dist 入口配合。
src/adapters/openclaw/index.ts 注册 kbprep_preflight、analyze、prepare、apply_review、cleanup、prepare_batch。 这是用户在 OpenClaw 中看到的工具表面。
src/worker.ts Node 调 Python worker，不起 HTTP 服务，不起 daemon。 决定超时、环境隔离、日志尾部、JSON envelope 是否稳定。
python/kbprep_worker/prepare.py 单文件主流程：诊断、转换、清洗、渲染、质量门、发布 latest。 当前最大业务核心，也已经被 known issues 标记需要拆小。
python/kbprep_worker/obsidian_kb.py 默认知识库策展层：去作者/营销/包装，保留知识正文。 这是“知识库不是营销稿”的核心实现。
python/kbprep_worker/quality.py 质量门：转换质量、清洗质量、细节保留、输出保留、图片引用。 防止假闭环，不能只靠肉眼看 cleaned.md。
python/kbprep_worker/cleanup.py 清理生命周期：finalize、expired、all。 现在需要支持 curated Obsidian 最终交付物。
python/kbprep_worker/prepare_batch.py 批量处理，sample-first，重文件串行，轻文件可并发。 批量结果也需要明确 curated 最终交付路径。
docs/known-issues.md 已知未完成项和 roadmap。 避免把已知债务伪装成已完成。
07 / benchmark

外部方法论对标

我对照了 Superpowers、Codex 插件机制、Claude Code 官方插件和当前 KBPrep。结论不是“谁更高级”，而是它们解决的是不同层级的问题：Superpowers 管开发纪律，Codex/Claude 插件管能力打包和分发，KBPrep 管真实业务处理引擎。

Superpowers

方法论层最强

它把 brainstorm、计划、TDD、调试、review、finish branch 拆成强制 skill 流程。适合防止 AI Coding 工具乱跑。

Codex 插件

产品包装层更完整

它强调 .codex-plugin/plugin.json、marketplace、interface metadata、default prompts、图标和截图，适合把能力变成可安装产品。

Claude Code 官方插件

生态分层最清楚

官方插件把 skills、commands、agents、hooks、MCP/LSP 分开，复杂流程交给 command + specialist agents，而不是塞进一个大函数。

KBPrep

业务引擎最实

当前仓库有真正的转换、OCR、质量门、审计、cleanup 和 CLI/OpenClaw adapter。问题是操作方法论和插件生态包装还不够成熟。

维度 Superpowers Codex 插件 Claude Code 官方插件 KBPrep 当前状态 该学谁
核心目标 让 agent 按正确开发流程工作。 把能力打包成可安装、可展示、可更新的 Codex 插件。 把技能、命令、agent、hook、MCP 组合成团队可复用能力包。 把本地资料转换、清洗、策展为知识库材料。 KBPrep 的开发治理学 Superpowers，分发包装学 Codex/Claude。
能力分层 每个流程是一项 skill，边界清楚。 manifest + skills/hooks/MCP/apps + marketplace。 commands 管入口，skills 管方法，agents 管专业角色，hooks 管强制动作，MCP 管外部工具。 工具入口完整，但业务大多集中在 Python worker，缺少 command/agent/hook 风格的操作层。 学 Claude 官方的职责拆分。
执行纪律 强制计划、TDD、review、验证。 偏插件生命周期和安装验证。 官方样例里 code-review 用多 agent + confidence filter，现代化插件用多个 specialist agents。 有测试和质量门，但缺少“每次处理资料时必须执行的 operator checklist”。 学 Superpowers 的 checklist 和 Claude code-review 的置信度过滤。
产品化程度 README 和 skill 结构成熟，但主要是方法论产品。 interface metadata、默认 prompt、图标、截图、marketplace policy 很完整。 官方 marketplace 有分类、来源、sha pinning、内部/外部插件分区。 有 README、docs、showcase，但没有 Codex/Claude 风格的 marketplace 产品包装。 学 Codex 的 interface metadata，学 Claude 的 marketplace trust model。
安全/信任 靠流程约束 agent，不直接解决第三方代码信任。 本地插件更新有 cachebuster 和 reinstall 规则。 官方文档强调信任插件，marketplace 插件会复制到本地 cache，路径用 plugin root 变量。 KBPrep 有 Python venv 隔离、PYTHONNOUSERSITE、质量门和 cleanup 防删边界。 KBPrep 的运行时隔离是优势；分发信任模型要学 Claude/Codex。
当前短板 对真实业务引擎没有现成实现。 不决定业务流程好坏。 官方插件体系强，但并不自动解决具体资料清洗质量。 默认 curated 模式交付和 cleanup finalize 口径未闭合；文档/skill 对 final_md 与 obsidian_dir 的描述冲突。 先补 KBPrep 自己的业务闭环，再吸收外部包装层。

更适合 KBPrep 的组合方案

  • 保留 KBPrep 当前 Python worker 作为业务核心，不要把转换/OCR/质量门改成纯 prompt 工作流。
  • 给 KBPrep 补一个 Superpowers 风格的 operator skill：样本先跑、看质量报告、看 discarded/review_needed、确认交付物、再 cleanup、最后 batch。
  • 把复杂审查拆成 Claude 官方风格 specialist agents：质量审计、营销噪音审计、细节保留审计、运行时发布审计。
  • 用 Codex 插件风格补产品包装：manifest、界面说明、默认 prompt、截图、安装/更新闭环。

不该照搬的东西

  • 不要把 KBPrep 变成只有 skills 的“提示词包”，因为它需要真实文件转换、OCR、运行时隔离和质量门。
  • 不要为了学官方插件就过早拆出一堆 agents；现在最重要的是默认交付/cleanup/batch 闭合。
  • 不要用 marketplace 包装掩盖业务断点。插件看起来专业，不等于用户能拿到正确结果。
  • 不要把所有验证都交给 AI review；确定性规则、测试和质量报告仍然是主裁判。

方法论结论

对 KBPrep 来说，最好的架构不是选择 Superpowers、Codex 插件或 Claude 官方插件其中之一，而是“三层合体”：底层保留 KBPrep 的确定性业务引擎；中层借鉴 Claude 官方插件的 commands/agents/hooks 分工；上层用 Superpowers 的计划、TDD、review、验收纪律约束 AI Coding；分发层再用 Codex/Claude marketplace 的 manifest 和产品化包装。

本次学习的主要来源

参考了 Superpowers 本地插件与公开 README、Codex 插件 manifest/marketplace 规范、Claude Code 官方 plugins / marketplace / reference 文档，以及 Anthropic 官方插件目录中的 claude-code-setup、code-review、code-modernization 样例。

08 / current runtime logic

当前运行逻辑

当前主入口是 kbprep_prepare(input_path, output_root)，默认 profile 是 curated_obsidian_kb。它先做普通清洗，再做知识库策展，最后由质量门决定是否发布 latest。

输入本地文件路径和 output_root。
诊断识别格式、PDF 子类型、文本层健康、是否需要 OCR。
转换直读、Office/EPUB/HTML、PDF text layer、MinerU/OCR。
清洗normalize、blockify、classify、clean rules、image clean。
策展curated Obsidian policy 去营销和身份包装。
质量门细节保留、输出保留、coverage、图片引用、坏文本层。
发布无 strict errors 才写 latest outputs。
通过时

发布 latest outputs

会把当前 run 的核心产物复制到 output_root 顶层，并在 curated 模式下发布 output_root/obsidian。

失败时

不更新 latest

如果 quality gate 有 strict errors，会返回 E_QA_FAILED，保留 run_dir 供检查。

争议点

final_md 与 obsidian 交付冲突

curated 模式故意让 final_md=null，避免把中间 cleaned.md 当最终 wiki note，但 cleanup 仍按 final_md 验收。

09 / ideal workflow

完整项目流程

完整流程不是“跑一次 prepare 成功”，而是从样本验证、质量审计、人工验收、清理收尾，到批量处理和版本发布的闭环。

代表样本先选一个真实文件，不直接批量。
preflight检查 Python、MinerU、PyMuPDF、torch、GPU/CPU。
analyze确认格式、路由和风险。
prepare单文件处理，默认 curated。
验收看 obsidian、quality、discarded、review_needed。
修正不满意就修规则或做 review patch。
finalize确认结果后清理过程材料。
batch多个样本通过后再批量。
1
单样本必须先过。不能一上来批量处理一堆 PDF，否则一个错误规则会污染整批输出。
2
最终交付物必须明确。standard 模式是源文件旁 Markdown；curated 模式应该明确是受保护的 Obsidian 目录或源文件旁知识库目录。
3
每个被删除的内容必须可审计。discarded.md 和 review_needed.md 不能被当成无意义日志。
4
质量门必须有权阻断发布。严格错误时不能更新 latest，也不能假装结果可交付。
5
cleanup 不能误删源文件和最终交付物，也不能因为默认模式没有 final_md 就无法 finalize。
6
OpenClaw runtime 必须验证。plugins list 不够，必须 inspect runtime，看 status、version、toolNames。
7
README、skill、CLI、测试必须讲同一个流程。否则后续 AI Coding 工具会沿着旧口径继续写错。
10 / current gap

当前最大断点

默认 curated 模式下，最终交付物从“源文件旁 Markdown”变成了“Obsidian folder”。这是合理的产品演进，但 cleanup、batch 和文档还没有完全跟上。

默认流程的结尾还没有完全闭合。

代码实际让 curated_obsidian_kb 的 latest_outputs.final_md 为 null，避免误把中间 cleaned.md 当最终知识库 note；但 cleanup finalize 仍要求 final_md 存在，这会让默认成功处理后的清理步骤卡住。

代码事实

prepare_artifacts.py 中 source_side_final = profile != "curated_obsidian_kb"，所以 curated 模式下 final_md 为空，主交付是 output_root/obsidian。

收尾事实

cleanup.py 的单文件 _finalize_single 仍读取 latest_outputs.final_md，没有就报 KBPREP_FINAL_OUTPUT_MISSING。

文档冲突

README 一处说日常用源文件旁 Markdown，另一处又说 curated 模式 Obsidian folder 是交付物且 final_md=null。两者需要分模式解释。

批量风险

prepare_batch 主要收集 batch_final_md，默认 curated 模式下可能没有这个字段，导致批量结果的“最终拿哪个文件/目录”不够清楚。
